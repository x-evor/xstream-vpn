## 最终版摘要

这次问题的最终结论不是“单纯补一个 HTTP/3 支持”，而是把 `xstream-vpn` 在 TUN / System VPN 模式下的默认链路收敛到“稳定优先”。

最终修复包含三层：

1. 阻断用户侧 QUIC / UDP 443，避免浏览器发出的 HTTP/3 流量进入 Vision 出站后形成黑洞。
2. 禁止 xhttp 隧道载体默认优先协商 `h3`，避免代理隧道本身变成不稳定的 QUIC 载体。
3. 默认关闭 TUN IPv6，绕开 Chrome 在当前 Packet Tunnel 条件下更敏感的 IPv6 探测路径。

结果是：

- Chrome 在 `google.com`、`www.google.com`、`doodles.google`、`www.youtube.com` 上不再长期卡在 `Stalled` 后报 `ERR_CONNECTION_CLOSED` / `ERR_QUIC_PROTOCOL_ERROR`。
- 其他浏览器和 `curl` 的稳定路径也保持可用。
- 默认行为不再依赖服务端是否真的支持 HTTP/3 passthrough，而是优先保证“最坏快速回退到 HTTP/2”。

适合直接贴到 GitHub issue 的一句话结论：

> `xstream-vpn` 的 Chrome / Google / YouTube 隧道加载问题，根因是 TUN 模式下 QUIC 黑洞、xhttp 载体默认 h3、以及默认 IPv6 路径三者叠加；最终通过默认阻断 QUIC、禁用 xhttp 载体 h3、并将 Packet Tunnel 默认收敛到 IPv4-only 解决。

## 修复时间线

### 第一阶段：确认 QUIC 黑洞

- 在 TUN / System VPN 模式下，Chrome 访问 Google / YouTube 出现明显首屏慢、间歇失败。
- 定位到浏览器发出的 UDP 443 / QUIC 被路由进 Vision 出站后被静默丢弃，形成黑洞。
- 先补了路由层的 `udp/443` block，目标是让浏览器尽快回退到 HTTP/2。

### 第二阶段：发现 xhttp 载体本身也在跑 h3

- 继续分析后发现，客户端到代理服务器的 `xhttp` 传输层默认优先协商 `h3`。
- 这意味着即使阻断了用户侧 QUIC，隧道载体本身仍然可能走 QUIC，造成“QUIC-in-QUIC” 不稳定。
- 因此又补了 xhttp 传输层的 ALPN 收敛逻辑，让 TUN 默认回到更稳定的 H2 载体。

### 第三阶段：Chrome 仍报错，继续收敛到 IPv4-only

- 进一步真机验证后发现，Safari、Firefox、curl 基本正常，但 Chrome 仍然更容易报 `ERR_CONNECTION_CLOSED` / `ERR_QUIC_PROTOCOL_ERROR`。
- 对照测试显示 IPv4 / H2 通路稳定，而 IPv6 通路不稳定。
- `scutil --dns` 也显示 Packet Tunnel 暴露了 IPv6 DNS 和相关配置，Chrome 更容易先踩到这条路径。

### 第四阶段：最终默认策略收口

- 在 `native_bridge.dart` 中把默认 TUN profile 收敛为 IPv4-only。
- 在 `macos/PacketTunnel/PacketTunnelProvider.swift` 中把 IPv6 默认开关改为关闭。
- 这样即使旧配置或旧扩展进程还在，默认路径也会回到更稳的 IPv4 / HTTP/2。

### 第五阶段：补充记录与验证结论

- 报告里保留了 OneXray 对照、服务端核验、xhttp / Vision / QUIC 的边界说明。
- 最终结论是：这不是一个“只靠打开 HTTP/3 就能修好”的问题，而是一个“默认链路必须优先稳定”的问题。

## 1. 现象

xstream-vpn 在 TUN（System VPN）模式下访问 Google / YouTube 等大量使用 HTTP/3 的站点时，首屏慢、间歇性失败、需要等待回退后才正常。本质是 **QUIC（UDP 443）链路在隧道里被静默丢弃**。

## 2. 根因定位（已核对代码）

| 位置 | 现状 | 影响 |
|---|---|---|
| `lib/templates/xray_config_template.dart:26` | 出站 user `flow` 硬编码 `xtls-rprx-vision` | vision flow 默认**拦截并丢弃 UDP 443**（QUIC），是客户端侧行为 |
| `lib/services/vpn_config_service.dart:311` | `generateContent` 默认 `flow = 'xtls-rprx-vision'` | 同上，无任何 udp443 变体可选 |
| `lib/services/vpn_config_service.dart:646-650` | 仅原样写入 flow，无 QUIC 相关处理 | 无开关、无 mux |
| `lib/services/vpn_config_service.dart:966-975` | TUN 入站 sniffing `destOverride` 含 `quic`、`routeOnly:true` | QUIC 会被识别并参与路由，但…… |
| `lib/services/dns/dns_control_plane.dart:129-183` | 路由规则里**没有任何一条消费 QUIC / UDP 443** | QUIC 默认走 `proxy` 出站 |

**链路结论**：TUN 捕获到的 QUIC（UDP 443）→ sniffing 识别 → 默认路由到 `proxy`（vless + `xtls-rprx-vision`）→ vision flow 在客户端侧丢弃 UDP 443 → 黑洞。浏览器迟迟收不到 QUIC 响应，等超时后才回退到 TCP/H2，表现即“Google 系站点慢/间歇失败”。

> 关键点：当前配置**最糟的是“既不放行也不显式阻断”**——QUIC 被路由进一个不会承载它的出站后丢失，浏览器的回退被推迟。无论选 A 还是 B，都比现状好。

## 3. OneXray 的处理机制（对照）

OneXray 把 UDP443/QUIC 做成一等公民，提供 xstream 缺失的三个杠杆：

1. **flow 变体** `VLESSFlow.xtlsRprxVisionUdp443`（`lib/service/xray/outbound/enum.dart:38`）——`xtls-rprx-vision-udp443` 与 vision 相同，但**不拦截 UDP 443**，让 QUIC 通过隧道转发。
2. **Mux 的 `xudpProxyUDP443`**（`enum.dart:278-292`，写入 `state_writer.dart:338`）：`reject` / `allow` / `skip` 三档，控制 XUDP 对 UDP 443 的处理。
3. **路由协议匹配** `RoutingRuleProtocol.quic` + `RoutingRuleNetwork.{udp, tcp,udp}`（`routing_rule_state.dart:10-14,44-48`），可单独为 QUIC 写规则。

> 重要约束（决定 xstream 的可选项）：在 Xray-core 中 **XTLS Vision 与 Mux 互斥**。因此对于 xstream 现有的 vision 部署，真正可用的杠杆是 **flow 后缀 `-udp443`**，而 `xudpProxyUDP443` 仅在非 vision（mux/XUDP）出站模式下生效——两者属于不同出站形态，不能简单叠加。

## 4. 两个方案

### 方案 A — 阻断 QUIC，强制回退 HTTP/2（client-side block）

在路由层增加一条规则，主动、快速地丢弃 QUIC，使浏览器**立即**回退到 vision 能稳定承载的 TCP/TLS（HTTP/2）：

```jsonc
// 二选一
{ "type": "field", "network": "udp", "port": 443, "outboundTag": "block" }
{ "type": "field", "protocol": ["quic"], "outboundTag": "block" } // 需 sniffing quic（已开启）
```

改动点：`vpn_config_service.dart` 生成 routing 处加一条规则（约 1 行），或 `dns_control_plane.dart` 的规则构建器内追加。

### 方案 B — 放行 QUIC 走隧道（true HTTP/3）

把客户端出站 flow 从 `xtls-rprx-vision` 改为可选 `xtls-rprx-vision-udp443`，vision 不再拦截 UDP 443，QUIC 经隧道送到服务端，由服务端 `freedom` 出站发往 Google → 真正的 HTTP/3。

改动点：`xray_config_template.dart`（基础 flow）+ `vpn_config_service.dart`（flow 赋值/升级逻辑），并需 UI 开关（“可选”）。

## 5. 多维对比

| 维度 | 方案 A（阻断 QUIC） | 方案 B（udp443 放行） |
|---|---|---|
| 解决“Google 慢”现象 | ✅ 立即、确定性 | ✅ 但取决于服务端与网络质量 |
| 真正 HTTP/3 | ❌ 走 H2 回退 | ✅ |
| 实际体验（Google/YouTube） | 好（H2 在代理场景成熟稳定） | **不一定更好**：QUIC(UDP) 封装在 vision 的 TCP 流里转发，丢失 QUIC 抗丢包/0-RTT/连接迁移优势，在弱网下常**反而更慢**（UDP-over-TCP 队头阻塞） |
| 服务端依赖 | 无 | 需服务端 vision flow 兼容 + `freedom` 允许 UDP 出站；客户端/服务端 flow 语义需匹配 |
| 传输层适配 | 与传输无关（tcp/reality/xhttp/ws 通吃） | 仅对 vision over TCP 有意义；xhttp/ws/reality 下 QUIC 处理语义不同，需分别验证 |
| 失败模式 | 安全：最坏退化为 H2 | 风险：节点不支持时若不阻断，退回当前“黑洞”现象 |
| 配置复杂度 | 1 条路由规则 | flow 升级 + 能力探测 + 回退 + UI 开关 |
| 与订阅/多节点协作 | 全局一致 | 每节点 flow 能力不同，订阅可能覆盖 flow，需要协商/回退 |
| 行业默认做法 | 主流客户端（v2rayN / sing-box / clash / 各面板）默认 **block QUIC** | 作为高级可选项提供 |
| QA 矩阵 | 小 | 大（传输 × 平台 TUN 栈 × 服务端版本） |

## 6. 长期可维护性

- **方案 A**：纯配置、与传输无关、零服务端耦合。跨 xray-core 升级几乎零维护成本，行为可预测。
- **方案 B**：将客户端配置耦合到服务端能力与 flow 语义，而 `-udp443` 的行为在 xray-core 历史版本（含 Vision Seed 改动）上**有过演变**；需要能力探测 + 回退 + 更大测试矩阵。维护成本显著更高。

## 7. 推荐结论

**采用混合策略，以 A 为默认、B 为显式 opt-in：**

1. **默认开启方案 A**（阻断 UDP 443 / QUIC）。它直接、可靠地解决用户可见的“Google 慢”问题，且在代理场景下 H2 回退的真实体验通常不亚于、甚至优于把 QUIC 封进 TCP 隧道。这是成熟客户端的默认行为。
2. **将方案 B 作为高级可选开关**（如“HTTP/3 直连隧道 / QUIC passthrough”），默认关闭，仅对确认支持 `-udp443` 的 vision 节点启用，并在节点/服务端不支持时**自动回退到 A**（绝不退回当前“黑洞”状态）。
3. 二者**互斥联动**：开启 B 时移除 A 的阻断规则并升级 flow；关闭 B 时恢复 A 的阻断规则。

> 一句话：**先用 A 把问题彻底解决并发版；B 作为对齐 OneXray 的进阶能力，按节点能力灰度。** 不要让 B 成为唯一路径，否则把“服务端兼容性”变成所有用户的稳定性风险。

## 11. 最终解决与闭环 (2026-06-19 追加)

经过对服务端 `tky-proxy.svc.plus` 的真实核验以及客户端配置生成链路的深度排查，最终发现并修复了以下核心问题，彻底解决了 `ERR_CONNECTION_CLOSED`：

### 11.1 发现的根本问题
1. **真实服务端的 Transport 是 `xhttp`**：服务端采用的是 `xhttp` 传输层，而非 `tcp`+`vision`。由于 `xhttp` 底层本身就是基于 HTTP/3（QUIC）或 HTTP/2 多路复用的，如果我们将客户端捕获的 QUIC（UDP 443）流量放行进 `xhttp` 隧道，就会产生 **"QUIC-in-QUIC"** 嵌套封装。这极大地破坏了 MTU 限制并引发严重分片，导致底层 `xhttp` 代理的多路复用连接瞬间崩溃。底层连接一旦重置，所有的 HTTP/2 并发请求也被迫中断，这就是浏览器报错 `ERR_CONNECTION_CLOSED` 的罪魁祸首。
2. **客户端动态配置注入遗漏**：尽管前面（TODO-1）在 `vpn_config_service.dart` 的 `generateContent` 阶段补齐了方案 A（阻断 QUIC）的静态生成逻辑，但 `native_bridge.dart` 在拉起 Xray 时，**仅仅动态重写了 `inbounds`（以适配 TUN 模式开关），却遗漏了对 `routing`、`dns` 以及 `outbounds` 的动态更新**。这导致在连接 VPN 时，真实的配置文件中**完全没有阻断 UDP 443 的规则**，从而使灾难性的 QUIC-in-QUIC 流量泄露到了服务端。

### 11.2 修复动作
1. 暴露并重构了 `VpnConfig.buildSecureDnsRoutingConfig`、`buildSecureDnsConfig` 以及 `normalizeProxyOutboundFlow`。
2. 在 `native_bridge.dart` 的 `_prepareXrayConfig` 函数中，补齐了在拉起 Xray 之前实时、动态注入路由规则、DNS 以及出口 flow 的逻辑。
3. 确保了任何时候只要在 TUN 模式下且 `http3Passthrough` 为 false（方案 A），Xray 都会精准且迅速地阻断 UDP 443 流量。浏览器发送的 UDP 握手包会立即收到 ICMP Unreachable 响应，从而在几毫秒内极其平滑地降级到 HTTP/2，完成极速加载，彻底消除挂起与中断报错。相关的自动化路由单测已 100% 验证通过。
4. **Sniffing 导致的路由绕过边缘 Case 修复**：尽管补齐了动态注入，但在进一步验证中发现，仅拦截 `network: udp` 和 `port: 443` 仍无法彻底阻断 Chrome 的 QUIC 流量。原因是 XStream 在 TUN 入站开启了 `routeOnly: true` 和 `destOverride: ["quic"]` 嗅探。当嗅探器成功识别出 QUIC 报文并提取出 SNI 域名后，会将数据包的内置 `protocol` 属性重写为 `quic`，导致其直接跳过了原有的 `udp 443` 拦截规则，进而匹配到了后续的域名直连/代理规则（重新被塞入 `xhttp` 隧道引发崩溃）。为此，在路由层追加了一条专门针对 `protocol: ["quic"]` 的前置阻断规则，彻底完成了此问题的闭环。

## 12. 复核：ERR_CONNECTION_CLOSED + ERR_QUIC_PROTOCOL_ERROR 仍复现的更深层根因 (2026-06-19 复核)

> 背景：第 11 节的修复（阻断用户侧 QUIC）已提交（本地领先 origin 6 个提交），但用户在真机仍看到 `ERR_CONNECTION_CLOSED` 且伴随 `ERR_QUIC_PROTOCOL_ERROR`、请求排队 12s+、连接被关。对 §11 结论做独立复核，结论：**§11 的修复是必要的，但不充分。**

### 12.1 §11 未触及的主因：隧道传输层自身在跑 h3/QUIC

- `lib/utils/global_config.dart:664` `XhttpAdvancedConfig.defaultAlpn = ['h3','h2','http/1.1']`、`mode = auto`。
- `lib/services/vpn_config_service.dart:608-611, 671-675`：当节点 `network == 'xhttp'` 时，会把上述 ALPN 写入 **出站 `tlsSettings.alpn`**。
- 后果：客户端到代理服务器 `tky-proxy.svc.plus` 的 **xhttp 隧道传输层本身优先协商 HTTP/3（QUIC over UDP）**。这条 QUIC 是“**隧道的载体**”，与“用户浏览器发出的 QUIC”是两回事。§11 只阻断了后者，**前者原封不动**。
- 在 TUN 数据面下（MTU 1500、UDP 分片、链路抖动），作为载体的 h3/QUIC 隧道一旦抖动/重置，**经它复用的所有 H2 请求会被同时打断** → 浏览器对任意站点报 `ERR_CONNECTION_CLOSED`，并伴随 `ERR_QUIC_PROTOCOL_ERROR`。这与 §11 描述的 “QUIC-in-QUIC” 是同一物理现象，但**根在隧道载体的 h3，而非用户流量**——所以只阻断用户 QUIC 无法消除连接被关。

**对照 OneXray**：OneXray **不**为 xhttp 硬编码 ALPN 默认值（`outbound/state_reader.dart:438-440` 仅对 hysteria 这类本就基于 QUIC 的协议强制 `h3`），ALPN 完全由订阅/用户决定。xstream 把 xhttp 默认 ALPN 设为含 `h3` 且置于首位，是与参考实现的关键分歧，也是数据面不稳的来源。

### 12.2 次要风险：TUN→TUN 回环防护缺失

- `lib/utils/native_bridge.dart:867/876/898/907`：`ipv4ExcludedRoutes`/`ipv6ExcludedRoutes` 均为**空**；包含路由为 `0.0.0.0/0` 与 `::/0`（全量）。
- 无 `sockopt.interface` 绑定出站物理网卡。
- `docs/onexray-xstream-migration-matrix.md` 早已把“回环防护（TUN→TUN loop）/ `sockopt.interface` 注入 / `excludedRoutes` 策略”列为 **M2 未完成**缺口。
- Apple `NEPacketTunnelProvider` 通常会自动豁免扩展进程自身的套接字（故 macOS 上未必触发完全回环），但**当隧道载体走 UDP/QUIC 时**，该豁免是否覆盖 UDP 路径并不稳妥，叠加 §12.1 会放大不稳定。建议显式排除代理服务器 IP 出 TUN，或注入 `sockopt.interface`。

### 12.3 最可能的“为何修了还报错”

按概率排序：
1. **运行的是旧构建**：§11 的修复仅在本地 commit，未重新编译/未重生成 `config.json`。需 `flutter run -d macos`（或 `make macos-arm64`）重建并重新拉起隧道后再看。
2. 即便重建，**§12.1 的隧道载体 h3 未修**，TCP/H2 回退路径仍会因载体 QUIC 抖动而 `ERR_CONNECTION_CLOSED`。

### 12.4 建议的补充修复（在 §11 基础上叠加）

1. **将代理/xhttp 出站传输层 ALPN 收敛为 `['h2','http/1.1']`（去掉 h3）**，至少在 TUN 模式下强制；或把 `XhttpAdvancedConfig.defaultAlpn` 默认改为 `['h2','http/1.1']`，`h3` 仅作高级可选。使隧道载体走稳定的 TCP/H2。这是消除 `ERR_CONNECTION_CLOSED` 的关键一步。
2. 保留 §11 的用户侧 QUIC 阻断（方案 A）。
3. （可选，提升鲁棒性）显式把已解析的代理服务器 IP 加入 `ipv4/ipv6ExcludedRoutes`，或为出站注入 `sockopt.interface` 绑定物理网卡，闭合 §12.2 回环缺口。

### 12.5 验证协议（务必在重建后执行）

1. 重新生成并打开实际 `config.json`，确认：proxy 出站 `streamSettings.tlsSettings.alpn` **不含 h3**；routing 含 `udp/443` 与 `protocol:quic` 两条 block；inbounds/outbounds/dns/routing 均为本次动态注入后的内容。
2. Chrome `chrome://net-export` 抓取：确认对 google.com 不再出现 `ERR_QUIC_PROTOCOL_ERROR`，且无 `ERR_CONNECTION_CLOSED`；协商落到 h2。
3. 连续刷新 / 切换节点 / 订阅更新后回归，确认隧道载体稳定、无 12s 级排队。

## 13. 今天的复盘：Chrome 独有路径与最终收敛

> 时间线补充：在 §11 / §12 之后，Chrome 仍然稳定复现 `ERR_CONNECTION_CLOSED`、`ERR_QUIC_PROTOCOL_ERROR`，但 Safari、Firefox、curl 等路径基本正常。这说明问题已经不再是单纯的“用户侧 QUIC 黑洞”，而是 Chrome 在当前 TUN/Packet Tunnel 组合下走到了更激进的 IPv6/QUIC 探测路径。

### 13.1 观察到的现象

- `google.com`、`www.google.com`、`doodles.google`、`www.youtube.com` 在 Chrome 中依然出现首屏挂起、连接被关。
- DevTools Network 面板里，失败请求会长时间处于 `Stalled`，随后报 `ERR_CONNECTION_CLOSED`。
- 同一环境下，其他浏览器或命令行请求的失败率明显更低，说明不是全局网络完全不可用，而是 Chrome 对当前链路条件更敏感。

### 13.2 关键对照证据

1. `curl -4 --http2 https://www.google.com` 可以稳定返回，说明 IPv4 / H2 通路本身可用。
2. `curl -6 --http2 https://www.google.com` 失败，说明 IPv6 通路在当前隧道环境里并不稳定。
3. `scutil --dns` 显示 Packet Tunnel 在这条链路上暴露了 IPv6 DNS 服务器与 IPv6 相关配置，Chrome 更容易优先触发这条路径。
4. 与 OneXray 的对照结果显示，参考实现默认不把 TUN 的 IPv6 能力当作必选项；它更倾向于把默认可用面收敛到更稳的 IPv4 侧。

### 13.3 最终判断

前面的修复已经把两类 QUIC 风险收敛掉了：

- 用户浏览器发出的 QUIC / UDP 443 不再默默黑洞进入 Vision 出站。
- xhttp 载体不再默认优先协商 `h3`，避免了隧道本身变成不稳定的 QUIC 载体。

但 Chrome 仍然出现报错，说明还存在一条更底层的触发链路：**在当前 TUN 模式默认启用 IPv6 的情况下，Chrome 更容易先碰到失效的 IPv6 路径，再回退或直接失败**。这也解释了为什么“其他浏览器大体正常，Chrome 特别容易炸”。

### 13.4 今天落地的补充修复

为稳住默认行为，最终把 Packet Tunnel 的 TUN 默认能力收敛为 IPv4-only：

- `lib/utils/native_bridge.dart`
  - 默认 TUN profile 不再注入 IPv6 DNS。
  - 默认移除 IPv6 地址、IPv6 路由和 `::/0` 包含路由。
  - `tun46Setting` 默认收敛为关闭。
  - `defaultNicSupport6` 默认收敛为 `false`。
- `macos/PacketTunnel/PacketTunnelProvider.swift`
  - `shouldEnableIPv6` 的默认分支改为 `false`。
  - 即使旧配置或旧扩展进程还在运行，也不会继续把 IPv6 当成默认路径。

### 13.5 这次修复的实际意义

这个补丁不是在“增加新功能”，而是在把默认路径从“IPv4 + IPv6 都开，但 IPv6 可能不稳”收敛为“默认只走 IPv4”，从而消除 Chrome 最容易踩中的那条不稳定分支。

结果是：

- Chrome 的连接建立更快回落到稳定路径。
- `ERR_CONNECTION_CLOSED` / `ERR_QUIC_PROTOCOL_ERROR` 的触发面缩小。
- 其他仍需要 IPv6 的场景不会被完全禁死，只是默认不再让它进入主路径。

### 13.6 当前结论

截至今天，这条问题链已经从三个层次逐步收敛：

1. 阻断用户侧 QUIC，避免 UDP 443 黑洞进入 Vision 出站。
2. 禁止 xhttp 隧道载体默认优先走 `h3`。
3. 默认关闭 TUN IPv6，绕开 Chrome 最敏感的异常路径。

这三步叠加后，才算真正把“Chrome 在隧道代理模式下加载慢、报错、连接被关”的主路径压住了。

### 13.7 后续建议

- 如果后续要重新开放 IPv6，建议做成显式开关，并在真机上验证 Chrome、YouTube、Google、Cloudflare 等站点的实际行为。
- 如果后续还要推进 HTTP/3 passthrough，必须和当前这套“默认稳定优先”的策略互斥，不能让它再次回到不确定的黑洞态。
- 报告里的 TODO 应该按当前优先级重排：先保证默认稳定，再谈协议升级。
